Superflix TV - Manual de instalacao do zero

Este manual foi escrito para tres publicos:

  1. **Comprador/cliente** que recebeu o projeto e quer instalar do zero.
  2. **Desenvolvedor humano** que vai trocar marca, Firebase, Cloudflare, Mercado Pago e gerar APK.
  3. **IA/coding assistant** que precisa entender o projeto sem quebrar os fluxos principais.
Importante: este projeto e uma base Android com catalogo, players externos, Firebase, Cloudflare Worker, AdMob e Mercado Pago. O comprador e responsavel por configurar suas proprias contas, chaves, dominio, politicas, direitos de uso e conformidade legal.

1. Estado atual do projeto

Funciona hoje

  • App Android nativo em Kotlin + Jetpack Compose.
  • Login/cadastro por Firebase Authentication.
  • Login com Google.
  • Firestore para usuarios, favoritos, historico, progresso, admin, reportes e assinatura Pro.
  • Catalogo via TMDB.
  • Filmes, series, animes e doramas.
  • Tela de detalhes com temporadas e episodios.
  • PLAYER 1 principal.
  • PLAYER 2, PLAYER 3 e PLAYER 4 opcionais.
  • Sistema de moedas com Reward Ads.
  • Plano Pro via Mercado Pago + Cloudflare Worker.
  • Painel admin dentro do app.
  • Painel web para revendedores.
  • Landing page para download.
  • Firebase Remote Config.
  • Firebase Cloud Messaging.
  • Picture-in-picture e player fullscreen.

Suspenso temporariamente

  • TV online/canais ao vivo.
  • Eventos esportivos.

Motivo: a fonte externa usada anteriormente caiu/ficou instavel. O app mostra mensagem de manutencao nessas areas.

Nao existe mais

  • Sistema tecnico de licenca dentro do app.

Motivo: foi removido para evitar custo, manutencao e risco de bloquear clientes legitimos. Para venda, use contrato/licenca comercial, entrega controlada e suporte pago.

4. Ordem recomendada de instalacao do zero

Siga nesta ordem para evitar retrabalho:

  1. Escolher nome do app, package name e cor principal.
  2. Criar projeto Firebase.
  3. Criar app Android no Firebase com o package name escolhido.
  4. Adicionar SHA-1/SHA-256 debug e release no Firebase.
  5. Baixar `google-services.json` e colocar no projeto.
  6. Ativar Authentication no Firebase.
  7. Criar Firestore em modo producao.
  8. Publicar regras do Firestore.
  9. Configurar Remote Config.
  10. Configurar Firebase Hosting para `player.html`.
  11. Criar service account do Firebase para o Worker.
  12. Criar conta/app no Mercado Pago.
  13. Configurar Cloudflare Worker.
  14. Configurar AdMob.
  15. Configurar TMDB API Key.
  16. Criar `local.properties`.
  17. Buildar APK debug e testar.
  18. Buildar APK release assinado.
  19. Testar em aparelho limpo.
  20. Atualizar landing page com APK final.

5. Personalizacao basica do app

5.1 Mudar nome do app

Arquivo:

app/src/main/AndroidManifest.xml

Altere:

<application
    android:label="Superflix TV"

Exemplo:

android:label="Meu Cine App"

Tambem procure textos de marca em:

app/src/main/java/com/superflixtv/app/MainActivity.kt
app/src/main/java/com/superflixtv/app/WelcomeAuthUi.kt
landing/index.html
reseller-panel/index.html

Busque por:

SUPERFLIX
Superflix
Superflix TV

Comando:

rg -n "SUPERFLIX|Superflix|Superflix TV" app landing reseller-panel

Pontos comuns:

  • Header do app: `MainActivity.kt`, funcao `Header`.
  • Tela inicial/login: `WelcomeAuthUi.kt`.
  • Nome no painel revendedor: `reseller-panel/index.html`.
  • Landing page: `landing/index.html`.

5.2 Mudar package name

Package atual:

com.superflixtv.app

Arquivos principais:

app/build.gradle.kts

Altere:

namespace = "com.superflixtv.app"
applicationId = "com.superflixtv.app"

Para, por exemplo:

namespace = "com.cliente.meuapp"
applicationId = "com.cliente.meuapp"

Aten??o:

  • O Firebase Android app precisa usar exatamente o mesmo package name.
  • O `google-services.json` precisa ser baixado novamente depois disso.
  • O Google Login depende desse package + SHA.
  • AdMob tambem deve usar o app correto.

Opcionalmente, renomeie as pastas Kotlin:

app/src/main/java/com/superflixtv/app

para:

app/src/main/java/com/cliente/meuapp

Se fizer isso, altere o package com.superflixtv.app em todos os .kt. Isso e mais trabalhoso. Para venda rapida, trocar apenas applicationId/namespace pode ser suficiente, mas o ideal profissional e renomear tudo.

6. Firebase do zero

6.1 Criar projeto Firebase

Tarefa humana no Firebase Console:

  1. Acesse Firebase Console.
  2. Clique em `Add project` / `Adicionar projeto`.
  3. Escolha o nome do projeto.
  4. Ative Google Analytics se quiser usar metricas.
  5. Crie o projeto.

6.2 Criar app Android no Firebase

Tarefa humana no Firebase Console:

  1. Dentro do projeto Firebase, clique no icone Android.
  2. Informe o package name do app.

Exemplo atual:

com.superflixtv.app
  1. Informe o apelido do app.
  2. Adicione SHA-1 e SHA-256.
  3. Baixe o arquivo `google-services.json`.

6.3 Onde colocar o `google-services.json`

O arquivo precisa ficar exatamente aqui:

app/google-services.json

Nao coloque em:

google-services.json
app/src/main/google-services.json
backend/google-services.json

Caminho correto:

Superflix/app/google-services.json

Depois de substituir:

  1. Android Studio: `File > Sync Project with Gradle Files`.
  2. Faca `Build > Clean Project`.
  3. Faca `Build > Rebuild Project`.
  4. Gere APK novo.

Se mudar package name ou SHA no Firebase, baixe outro google-services.json.

6.4 SHA-1 e SHA-256

Para Google Login funcionar, o Firebase precisa conhecer a assinatura do APK.

Comando:

$env:JAVA_HOME="$env:ProgramFiles\Android\Android Studio\jbr"
.\gradlew.bat :app:signingReport --no-daemon

Cadastre no Firebase:

  • SHA-1 debug, para testes pelo Android Studio/debug APK.
  • SHA-256 debug, recomendado.
  • SHA-1 release, para APK distribuido.
  • SHA-256 release, para APK distribuido e Play Integrity.

Se publicar na Play Store, tambem cadastre o SHA da Play App Signing.

6.5 Ativar Authentication

Tarefa humana no Firebase Console:

  1. Va em `Authentication`.
  2. Clique em `Get started`.
  3. Ative `Email/Password`.
  4. Ative `Google`.
  5. Em Google, selecione email de suporte.
  6. Salve.

Se o login Google mostrar erro 10::

  • SHA errado ou ausente.
  • `google-services.json` antigo.
  • APK buildado antes de baixar o novo JSON.
  • Package name diferente.

Corre??o:

  1. Adicione SHA correto no Firebase.
  2. Baixe novo `google-services.json`.
  3. Substitua em `app/google-services.json`.
  4. Clean/Rebuild.
  5. Instale APK novo.

6.6 Criar Firestore

Tarefa humana no Firebase Console:

  1. Va em `Firestore Database`.
  2. Clique em `Create database`.
  3. Escolha `Production mode`.
  4. Escolha regiao.
  5. Crie.

Depois publique as regras do projeto.

Com Firebase CLI:

firebase login
firebase use --add
firebase deploy --only firestore:rules,firestore:indexes

Se firebase nao existir:

npm install -g firebase-tools
firebase login

Alternativa sem instalar global:

npx firebase-tools login
npx firebase-tools deploy --only firestore:rules,firestore:indexes

6.7 Criar usuario admin

Depois de criar uma conta no app, pegue o UID do usuario no Firebase Authentication.

No Firestore, crie colecao:

admins

Documento:

admins/{UID_DO_USUARIO}

Campos:

enabled: true       boolean
name: "Nome"        string
role: "owner"       string

Exemplo:

admins/NUX1IgH8ytgMxON6rmbuOS7y1bu2
  enabled: true
  name: "Mateus"
  role: "owner"

Importante:

  • Colecao e `admins`, plural.
  • `enabled` e boolean, nao string.
  • `role` e string.

Depois disso, feche e abra o app. A aba Admin deve aparecer.

6.8 Remote Config

Tarefa humana no Firebase Console:

  1. Va em `Remote Config`.
  2. Crie os parametros abaixo.
  3. Publique as alteracoes.

Parametros usados pelo app:

minimum_version           number   Ex: 1
unlock_cost               number   Ex: 2
unlock_hours              number   Ex: 6
missions_enabled          boolean  true/false
banner_enabled            boolean  true/false
download_url              string   URL da pagina/APK
pro_price                 string   Ex: 5,99
global_message            string   Aviso opcional na Home
report_player_enabled     boolean  true/false

Uso:

  • `minimum_version`: se for maior que `versionCode`, o app obriga atualizar.
  • `download_url`: link usado na tela de atualizacao obrigatoria.
  • `global_message`: mostra aviso na Home.
  • `pro_price`: aparece na Central Pro.

6.11 App Check / Play Integrity

O projeto tem App Check:

  • Debug usa provider debug.
  • Release usa Play Integrity.

Arquivos:

app/src/debug/java/com/superflixtv/app/AppCheckProvider.kt
app/src/release/java/com/superflixtv/app/AppCheckProvider.kt

Para testes fora da Play Store, cuidado com enforcement do App Check. Se ativar enforcement antes de configurar tudo, pode quebrar Firestore/Auth/Remote Config.

Recomendacao:

  1. Durante testes, nao force enforcement.
  2. Configure SHA-256 corretamente.
  3. Se for usar Play Store, configure Play Integrity/App Check com calma.
  4. So depois ative enforcement.

No debug, o Logcat pode mostrar um token como:

Enter this debug secret into the allow list...

Adicione esse token no Firebase App Check se quiser testar com enforcement.

7. Google Services JSON - problemas comuns

Problema: login Google mostra `10:`

Causa quase sempre e uma destas:

  • SHA-1 nao adicionado.
  • SHA-256 nao adicionado.
  • `google-services.json` antigo.
  • Package name diferente do Firebase.
  • APK foi buildado antes de atualizar o JSON.

Resolucao:

  1. Gere `signingReport`.
  2. Adicione SHA no Firebase.
  3. Baixe novo `google-services.json`.
  4. Substitua em `app/google-services.json`.
  5. Clean/Rebuild.
  6. Instale APK novo.

Problema: `default_web_client_id` ausente

Abra app/google-services.json e procure:

oauth_client
client_type: 3

Se nao existir, o Google Sign-In pode falhar. Normalmente baixar o JSON depois de ativar Google Login e adicionar SHA resolve.

8. Cloudflare Worker do zero

O Worker cuida de:

  • Assinatura Pro via Mercado Pago.
  • Webhook Mercado Pago.
  • Painel revendedor.
  • Pos-pago de revendedores.
  • PIX para divida pos-paga.
  • Endpoint `/content/trending` para landing.

Arquivo principal:

cloudflare-worker/src/index.ts

Config:

cloudflare-worker/wrangler.toml

8.1 O que o humano faz no Cloudflare

  1. Criar conta Cloudflare.
  2. Instalar/logar Wrangler:
cd cloudflare-worker
npx wrangler login
  1. Conferir `wrangler.toml`.
  2. Definir secrets.
  3. Deployar Worker.
  4. Copiar URL final do Worker.
  5. Colocar a URL no app via `local.properties`.
  6. Colocar a URL no webhook do Mercado Pago.

8.2 Variaveis em `wrangler.toml`

Edite:

name = "superflix-subscriptions"
main = "src/index.ts"
compatibility_date = "2026-07-12"

[vars]
FIREBASE_PROJECT_ID = "seu-projeto"
FIREBASE_API_KEY = "sua-api-key-web/firebase"
APP_RETURN_URL = "https://seu-projeto.web.app/subscription-return.html"
PLAN_TITLE = "Nome do App Premium"
PLAN_PRICE = "5.99"
RESELLER_POSTPAID_CREDIT_PRICE = "2.99"

Pode mudar name para o nome do cliente:

name = "meuapp-subscriptions"

A URL final fica parecida com:

https://meuapp-subscriptions.SEUSUBDOMINIO.workers.dev

8.3 Secrets do Worker

Nunca coloque esses valores diretamente no wrangler.toml:

MERCADOPAGO_ACCESS_TOKEN
MERCADOPAGO_WEBHOOK_SECRET
FIREBASE_CLIENT_EMAIL
FIREBASE_PRIVATE_KEY

Configure assim:

cd cloudflare-worker
npx wrangler secret put MERCADOPAGO_ACCESS_TOKEN
npx wrangler secret put MERCADOPAGO_WEBHOOK_SECRET
npx wrangler secret put FIREBASE_CLIENT_EMAIL
npx wrangler secret put FIREBASE_PRIVATE_KEY

Se o Worker usa TMDB no endpoint de trending, configure tambem:

npx wrangler secret put TMDB_API_KEY

Ou coloque como var, mas secret e mais seguro.

8.4 Onde pegar `FIREBASE_CLIENT_EMAIL` e `FIREBASE_PRIVATE_KEY`

Tarefa humana no Firebase/Google Cloud:

  1. Firebase Console.
  2. Configuracoes do projeto.
  3. Aba `Service accounts` / `Contas de servico`.
  4. Clique em gerar nova chave privada.
  5. Baixe o JSON.

No JSON:

{
  "client_email": "firebase-adminsdk-xxxxx@seu-projeto.iam.gserviceaccount.com",
  "private_key": "-----BEGIN PRIVATE KEY-----\n...\n-----END PRIVATE KEY-----\n"
}

Use:

FIREBASE_CLIENT_EMAIL = client_email
FIREBASE_PRIVATE_KEY = private_key

No wrangler secret put FIREBASE_PRIVATE_KEY, cole a chave inteira, incluindo:

-----BEGIN PRIVATE KEY-----
...
-----END PRIVATE KEY-----

Se der erro por quebra de linha, substitua quebras reais por \n ou cole exatamente como o terminal aceitar.

8.5 Deploy do Worker

cd cloudflare-worker
npm install
npx tsc --noEmit --target ES2022 --module ESNext --moduleResolution Bundler --lib ES2022,WebWorker src/index.ts
npx wrangler deploy

Teste:

Invoke-WebRequest https://SEU-WORKER.workers.dev/health

Resposta esperada:

{"ok":true}

8.6 Colocar Worker no app

Arquivo local, nao versionado:

local.properties

Adicione:

Depois:

  1. Sync Gradle.

  3. Gere APK novo.

buildConfigField("String", "ADMOB_REWARDED_ID", "\"ca-app-pub-3940256099942544/5224354917\"")

Pelo ID real:

buildConfigField("String", "ADMOB_REWARDED_ID", "\"ca-app-pub-XXXXXXXXXXXXXXXX/ZZZZZZZZZZ\"")

Durante desenvolvimento, mantenha IDs de teste para evitar trafego invalido.

14. Build release assinado

Pelo Android Studio:

  2. Escolha APK.

APK geralmente fica em:

Ou, se gerado pelo Android Studio:

25. Guia especifico para agentes de IA como Codex, Claude, Cursor e Copilot

Esta secao define claramente o que um agente de IA pode fazer sozinho no projeto e o que precisa ser feito por um humano. Use isto como instrucao operacional para qualquer IA que receber este codigo-fonte.

25.1 Papel esperado da IA

Um agente de IA pode atuar como desenvolvedor assistente do projeto. Ele pode:

  • Ler e entender a estrutura do codigo.
  • Explicar arquitetura e fluxos.
  • Editar arquivos locais do projeto.
  • Corrigir bugs de Kotlin/Compose.
  • Corrigir bugs de TypeScript do Cloudflare Worker.
  • Atualizar documentacao.
  • Refatorar componentes.
  • Criar novas telas.
  • Criar novos ViewModels.
  • Ajustar layout, textos, cores e marca.
  • Adicionar ou remover dependencias Gradle, com cuidado.
  • Alterar templates de player no codigo.
  • Preparar instrucoes para Firebase, Cloudflare, Mercado Pago, TMDB e AdMob.
  • Rodar comandos locais de build/teste quando tiver permissao.

Um agente de IA nao deve assumir que tem acesso real as contas do comprador. Mesmo que consiga editar arquivos, algumas tarefas dependem de a pessoa humana entrar em pain?is externos.
